## 一、项目概述

这是一个面向高校寝室的 **AI 智能签到系统**。学生可以通过浏览器完成：
- 🔍 **人脸识别 + GPS 定位** 双重验证签到
- 💬 **AI 智能对话** 查询签到记录、请假报备、寝室规则
- 📋 **请假申请** 自动收集信息并提交审核

核心亮点是将 **大语言模型（DeepSeek）** 与寝室管理业务深度结合，通过 LangChain4j 框架实现意图识别、工具调用、结构化输出和 RAG 知识库检索。

## 三、功能模块详解

### 3.1 用户认证模块

**文件**：`AuthController.java` + `AuthServiceImpl.java`

#### 登录流程
```
用户输入学号+密码 → BCrypt 验证 → Session 存储用户信息
                                    ↓
                              可选：记住我（Cookie + SHA-256 Token）
```

#### 密码安全
- 使用 **BCrypt** 加密存储密码（`BCryptPasswordEncoder`），数据库中不存明文
- 不从 Spring Security 全栈引入，只引入 `spring-security-crypto` 模块，保持轻量

#### "记住我"功能
```
勾选"记住我" →
  1. 生成 UUID 随机 token
  2. SHA-256(token) 存入 remember_token 表
  3. 原始 token 写入 Cookie（HttpOnly + Secure + SameSite=Lax）
  4. Cookie 有效期 30 天
```

**为什么用 SHA-256 而不是存原始 token？**
> 即使数据库泄露，攻击者也无法用 hash 反推出 Cookie 中的原始 token 来伪造登录。

#### Session 管理
- Session 超时 30 分钟（`server.servlet.session.timeout=30m`）
- 登出时同时清理 Session 和数据库中的 token

### 3.2 AI 智能对话模块

**文件**：`AiController.java` + `AiChatServiceImpl.java`

这是系统的**核心模块**，实现了从用户消息到智能回复的完整链路。

#### 整体流程图

```
用户消息
    │
    ▼
┌─────────────┐
│  意图识别    │ ← AiAssistant (DeepSeek 分析意图)
│  intention   │   返回 JSON: {intention: 1-5, output: "..."}
└──────┬──────┘
       │
       ├── intention=1 → 请假报备 (LeaveRequestAssistant)
       ├── intention=2 → 状态查询 (QueryCheckInAssistant)  
       ├── intention=3 → 规则问答 (RAG 知识库检索)
       ├── intention=4 → 闲聊 (直接返回 output)
       └── intention=5 → 个人信息 (直接返回 output)
```

#### 意图分类（5 类）

`getIntention.txt` 是意图识别的 System Prompt，定义了 5 种意图：

| 意图编号 | 说明 | 示例 |
|----------|------|------|
| 1 | 请假报备 | "我要请假"、"今晚不回寝室" |
| 2 | 状态查询 | "我签到了吗"、"我的请假审批了吗" |
| 3 | 规则问答 | "晚归几点算违纪"、"寝室几点关门" |
| 4 | 闲聊/其他 | "你好"、"今天天气怎么样" |
| 5 | 个人信息 | "我是谁"、"我的学号是多少" |

#### AI Service 定义（LangChain4j `@AiService`）

每个 AI 助手都是一个 **接口**，LangChain4j 在运行时自动生成代理实现：

```java
// 意图识别助手
@AiService(chatModel = "openAiChatModel", tools = {"chatHistoryTools"})
public interface AiAssistant {
    @SystemMessage(fromResource = "getIntention.txt")
    IntentionOutput intention(@V("sessionId") String sessionId, ...);
}

// 签到登记助手
@AiService(chatModel = "openAiChatModel", tools = {"chatHistoryTools", "checkInRecordTools"})
public interface CheckInAssistant {
    @SystemMessage(fromResource = "checkin.txt")
    CheckInOutput checkIn(@V("sessionId") String sessionId, ...);
}

// 状态查询助手（流式输出）
@AiService(streamingChatModel = "openAiStreamingChatModel", 
           tools = {"chatHistoryTools", "checkInRecordTools", "leaveRequestTools"})
public interface QueryCheckInAssistant {
    Flux<String> queryCheckIn(...);  // 返回 Flux 流式输出
}
```

关键设计：
- `@SystemMessage(fromResource = "...")` — 从 classpath 加载 System Prompt 文本文件
- `@V("variableName")` — 模板变量注入，如 `{{studentId}}`、`{{message}}`
- `tools = {...}` — 声明 LLM 可以调用的工具（Bean 名称）
- `IntentionOutput` / `CheckInOutput` — **结构化输出**，LLM 直接返回 JSON 后被反序列化为 Java 对象

### 3.3 RAG 知识库检索模块

**文件**：`RagConfig.java` + `AiChatServiceImpl.embeddingQuery()`

这是系统中最关键的技术点之一：**如何让 AI 回答寝室管理规定？**

#### RAG 是什么？

> RAG（Retrieval-Augmented Generation，检索增强生成）的核心思路是：**先检索相关知识，再让 AI 基于检索结果回答**。这样 AI 不需要在训练时"记住"所有规定，而是实时查询知识库。

#### 本系统的 RAG 实现流程

```
┌─────────────────────────────────────────────────────────┐
│                    1. 知识库构建阶段                       │
│                                                         │
│  寝室规则文本 → 按行切分 → 向量化(Embedding) → 存入向量库  │
│                                                         │
│  例："寝室关门时间：每晚23:00，晚归记为违纪"                  │
│       → DashScope text-embedding-v3 向量化                │
│       → [0.023, -0.451, 0.789, ...] (1024维向量)          │
│       → 存入 InMemoryEmbeddingStore                       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    2. 用户查询阶段                         │
│                                                         │
│  用户问题："几点关寝室门？"                                  │
│       │                                                  │
│       ▼                                                  │
│  向量化用户问题 → [0.018, -0.432, 0.801, ...]             │
│       │                                                  │
│       ▼                                                  │
│  在向量库中搜索最相似的 2 条规则（minScore=0.8）             │
│       │                                                  │
│       ▼                                                  │
│  返回匹配规则 → 拼成回答返回用户                            │
│  "根据寝室管理规定：                                         │
│   1. 寝室关门时间：每晚23:00，晚归记为违纪                    │
│   2. 晚归三次以上给予警告处分"                               │
└─────────────────────────────────────────────────────────┘
```

#### 代码实现详解

**1) 向量模型配置（RagConfig.java）**
```java
@Bean
public EmbeddingModel embeddingModel() {
    return OpenAiEmbeddingModel.builder()
        .baseUrl("https://dashscope.aliyuncs.com/compatible-mode/v1")  // 阿里云 DashScope
        .apiKey("sk-...")
        .modelName("text-embedding-v3")   // 阿里云 Embedding 模型
        .timeout(Duration.ofMinutes(1))
        .build();
}
```
> **为什么用 DashScope 而不是 DeepSeek？** — 因为 DeepSeek 不提供 Embedding API。这里使用阿里云的 `text-embedding-v3` 模型，通过 OpenAI 兼容接口调用。

**2) 向量存储（InMemoryEmbeddingStore）**
```java
@Bean
public EmbeddingStore<TextSegment> embeddingStore() {
    return new InMemoryEmbeddingStore<>();  // 内存存储，重启后需重新构建索引
}
```
> ⚠️ **注意**：目前使用的是内存存储，服务重启后知识库索引会丢失，需要重新调用 `/ai/embedding-index` 接口构建。

**3) 知识库构建（embeddingIndex）**
```java
public String embeddingIndex() {
    // 1. 定义寝室规则
    List<String> rules = List.of(
        "寝室关门时间：每晚23:00，晚归记为违纪",
        "晚归三次以上给予警告处分",
        "夜不归宿一次给予严重警告处分",
        "寝室内禁止使用大功率电器（超过800W）",
        "请假需提前一天向辅导员提交申请",
        "病假需提供医院证明",
        "未经批准擅自离校按旷课处理",
        "每周日晚进行寝室卫生检查"
    );
    
    // 2. 写入临时文件
    FileUtil.appendLines(rules, path, StandardCharsets.UTF_8);
    
    // 3. 加载文档 → 按行切分（每段200字符，重叠100字符）
    DocumentSplitter splitter = new DocumentByLineSplitter(200, 100);
    List<TextSegment> segments = splitter.split(document);
    
    // 4. 批量向量化 → 存入向量库
    List<Embedding> embeddings = embeddingModel.embedAll(segments).content();
    embeddingStore.addAll(embeddings, segments);
}
```

**4) 相似度检索（embeddingQuery）**
```java
public List<String> embeddingQuery(String message) {
    EmbeddingSearchRequest request = EmbeddingSearchRequest.builder()
        .queryEmbedding(embeddingModel.embed(message).content())  // 向量化用户问题
        .maxResults(2)         // 最多返回 2 条最相似规则
        .minScore(0.8)         // 相似度阈值 0.8
        .build();
    EmbeddingSearchResult<TextSegment> result = embeddingStore.search(request);
    return result.matches().stream()
        .map(x -> x.embedded().text())
        .collect(Collectors.toList());
}
```

### 3.4 人脸识别 + GPS 签到模块

**文件**：`CheckInController.java` + `BaiduFaceService.java` + `LocationVerificationServiceImpl.java`

签到需要 **双重验证**：人脸识别 + GPS 定位。

#### 人脸识别流程

```
┌────────────────────────────────────────────┐
│            FaceRegisterController           │
│                                            │
│  POST /face/detect                         │
│    1. 接收多帧图片（frames[]）               │
│    2. 逐帧活体检测（防照片攻击）              │
│    3. 第一帧人脸搜索匹配                     │
│    4. 验证匹配结果 == 当前登录学生ID          │
│                                            │
│  POST /face/register                       │
│    1. 检测图片中是否有人脸                    │
│    2. 注册到百度人脸库 (group_id=campus)     │
│    3. 更新学生表 face_image_url = "registered"│
└────────────────────────────────────────────┘
```

#### 百度 AI 人脸识别 API

系统接入了百度 AI 的 3 个核心 API：

| API | 用途 | 关键参数 |
|-----|------|---------|
| `/face/v3/detect` | 人脸检测 + 活体检测 | `face_field=spoofing` 获取活体分数 |
| `/face/v3/faceset/user/add` | 人脸注册 | `group_id=campus_checkin` |
| `/face/v3/search` | 人脸搜索匹配 | `match_threshold=80`（相似度阈值80%） |

#### 多帧活体检测（防照片攻击）

```java
public LivenessResult verifyLivenessMultiFrame(List<String> base64Images) {
    int passCount = 0;
    for (int i = 0; i < min(base64Images.size(), 2); i++) {
        if (i > 0) Thread.sleep(2000);  // QPS 限制：帧间间隔2秒
        LivenessResult frameResult = verifyLiveness(base64Images.get(i));
        if (frameResult.isSuccess()) passCount++;
    }
    // 2帧中至少1帧通过即认为活体
    boolean overallSuccess = passCount >= 1;
}
```

**活体检测原理**：
- 百度 API 返回 `spoofing` 值（0-1），表示"照片攻击"的概率
- 本系统阈值：`spoofing < 0.7` 认为活体（真人），≥0.7 认为可能是照片/屏幕翻拍

#### GPS 定位验证（Haversine 公式）

```java
// 配置的寝室坐标（浙江国际海运职业技术学院）
app.dorm.latitude=29.991316
app.dorm.longitude=122.179503
app.dorm.radius=500  // 允许误差 500 米

// Haversine 公式计算两点间距离
private double haversine(double lat1, double lon1, double lat2, double lon2) {
    double dLat = Math.toRadians(lat2 - lat1);
    double dLon = Math.toRadians(lon2 - lon1);
    double a = sin(dLat/2)² + cos(lat1)*cos(lat2)*sin(dLon/2)²;
    double c = 2 * atan2(√a, √(1-a));
    return 6371000 * c;  // 地球半径 × 弧度 = 距离（米）
}
```

### 3.5 聊天历史自动记录（AOP 切面）

**文件**：`ChatFlowAop.java` + `ChatFlow.java`

```java
@Around("@annotation(org.example.aop.ChatFlow)")
public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
    // 1. 在 AI 方法执行前，保存用户消息到 chat_history 表
    saveChatHistory(sessionId, message, "0");  // role=0 表示用户
    
    // 2. 执行 AI 方法
    Object result = joinPoint.proceed();
    
    // 3. 保存 AI 回复到 chat_history 表
    saveChatHistory(sessionId, result.toString(), "1");  // role=1 表示AI
    
    return result;
}
```

**设计巧妙之处**：
- 使用 Spring AOP 在 `chatStream` 方法上织入切面
- AI 方法本身不需要关心历史记录的保存逻辑
- `ChatHistoryTools.getChatHistory()` 作为 LangChain4j Tool 暴露给 LLM，让 AI 能"记住"之前的对话

## 五、遇到的问题与解决方案

### 问题 1：DeepSeek 不提供 Embedding API

**问题**：RAG 需要文本向量化，但 DeepSeek 只有 Chat API，没有 Embedding API。

**解决**：使用阿里云 DashScope 的 `text-embedding-v3` 模型，通过 OpenAI 兼容接口调用。LangChain4j 的 `OpenAiEmbeddingModel` 天然支持兼容模式：
```java
OpenAiEmbeddingModel.builder()
    .baseUrl("https://dashscope.aliyuncs.com/compatible-mode/v1")
    .modelName("text-embedding-v3")
```

---

### 问题 2：RAG 检索结果质量不高

**问题**：直接用向量相似度检索寝室规则时，经常返回不太相关的内容。比如用户问"请假怎么申请"，可能返回"晚归处分"相关的规则。

**解决策略**：
1. **意图前置分类**：在 RAG 检索之前，先用 `AiAssistant` 做意图识别。只有 `intention=3`（规则问答）才走 RAG 检索，其他意图走对应处理逻辑
2. **相似度阈值**：设置 `minScore=0.8`，过滤低相关性结果
3. **知识库条目精心设计**：每条规则都是一句完整、独立的话，不依赖上下文

---

### 问题 3：百度人脸 API 免费版 QPS 限制

**问题**：百度 AI 免费版 QPS（每秒请求数）限制约 1 次/秒，多帧活体检测时连续请求会导致 `Open api qps request limit reached` 错误。

**解决方案**：
1. **帧间延迟**：每帧之间 `Thread.sleep(2000)` 间隔 2 秒
2. **帧数限制**：只取前 2 帧做活体检测（而非全部 5 帧）
3. **搜索前等待**：活体检测结束后再等 2 秒才进行人脸搜索
4. **多数投票**：2 帧中至少 1 帧通过即认为活体（宽松策略）

```java
// 关键代码
for (int i = 0; i < Math.min(base64Images.size(), 2); i++) {
    if (i > 0) Thread.sleep(2000);  // QPS 限制
    LivenessResult frameResult = verifyLiveness(base64Images.get(i));
    if (frameResult.isSuccess()) passCount++;
}
Thread.sleep(2000);  // 搜索前等待 QPS 恢复
```

### 问题 4：结构化输出不稳定

**问题**：早期让 LLM 自由输出 JSON 时，经常出现格式错误（多一个逗号、少一个引号、JSON 外多了说明文字），导致 `IntentionOutput` 反序列化失败。

**解决方案**：
1. Prompt 中明确写死 `"输出格式（严格JSON，不要添加任何其他文字）"`
2. 提供完整的 JSON 示例（不是描述，是直接贴示例）
3. 使用 LangChain4j 的 `@Description` 注解给每个字段添加描述，框架会自动在 Prompt 中注入字段说明
4. Prompt 末尾的示例展示了每种意图的期望输出

### 问题 5：前端跨页面跳转时 Session 丢失

**问题**：从首页跳转到签到页面（`/checkin`）时，有时 Session 丢失导致"未登录"错误。

**原因**：Cookie 的 `SameSite` 属性导致跨页面请求不带 Cookie。

**解决方案**：
1. 在 `AuthController.login` 中，`Set-Cookie` 响应头显式设置 `SameSite=Lax`
2. 前端使用 `fetch` 时不需要额外配置，浏览器会自动携带同站 Cookie
3. PageController 做 Session 检查，未登录则重定向到登录页

### 问题 6：前端暗色模式适配

**问题**：如何支持亮色/暗色/跟随系统三种模式？

**解决方案**：纯 CSS 变量 + 少量 JS：
```css
:root, [data-theme="light"] { /* 亮色变量 */ }
[data-theme="dark"] { /* 暗色变量 */ }
@media (prefers-color-scheme: dark) {
    :root:not([data-theme="light"]) { /* 跟随系统暗色 */ }
}
```

通过 `html` 元素上的 `data-theme` 属性切换，JS 只负责读写 `localStorage` 和设置属性。

### 问题 7：AI 对话中存在大量"废话"回复

**问题**：LLM 在不该长篇大论时给了很长的回复，用户体验差。

**解决方案**：**意图分离 + 渐进式引导**
1. 规则问答（intention=3）不经过 LLM 生成，直接用 RAG 检索结果拼接固定模板
2. 签到登记（intention=1 实际被注释掉了）和请假报备有精炼的 System Prompt，要求 LLM 逐步收集信息、一次只问 1-2 个问题
3. 闲聊（intention=4）限制在"寝室相关"话题，拒绝回答无关问题

### 问题 8：Cloudflare Tunnel 断开导致 502 Bad Gateway

**问题**：网站访问时提示 `502 Bad Gateway`（Cloudflare 错误页），但本地服务正常运行。

**原因**：本系统通过 Cloudflare Tunnel (`cloudflared`) 将本地 8088 端口暴露到公网。当 `cloudflared` 进程因电脑重启/网络波动等原因停止后，Cloudflare 无法连接到后端，就会返回 502。

**解决方案**：
1. **重新启动 cloudflared 隧道**（如果有安装）：
   ```bash
   cloudflared tunnel run <tunnel-name>
   ```
2. **备选方案 — 路由器端口转发**：在路由器中设置端口转发，将公网 443/8088 转发到本机 8088，同时确保 Cloudflare DNS 的 A 记录指向当前公网 IP
3. **检查 Windows 防火墙**：确保入站规则允许 8088 端口
4. **开机自启**：将 cloudflared 注册为 Windows 服务，避免电脑重启后手动启动

**排查命令**：
```bash
# 检查本机公网 IP
curl ifconfig.me

# 检查 cloudflared 是否在运行
tasklist | findstr cloudflared

# 检查 8088 端口是否监听
netstat -ano | findstr :8088
```

## 六、校园统一身份认证集成方案

> **状态：规划中**。当前系统使用学号+BCrypt 密码的自主登录。以下是接入学校统一身份认证平台的完整方案。

### 6.1 为什么要接入校园统一认证？

| 对比 | 当前方案 | 统一认证方案 |
|------|---------|-------------|
| 账号管理 | 需要手动导入学生账号 | 复用学校已有的账号体系 |
| 密码安全 | 自己存 BCrypt 密码 | 密码由学校信息中心管理，系统不接触密码 |
| 用户体验 | 学生需要记额外的密码 | 学生用统一身份认证账号（学号+校园密码）登录 |
| 合规性 | 自建账号体系 | 符合高校信息化建设规范 |

### 6.2 主流校园认证协议

中国高校常见的统一身份认证协议有三种：

| 协议 | 说明 | 典型场景 |
|------|------|---------|
| **CAS** | Central Authentication Service，耶鲁大学开源 | 老牌高校常用（如浙大、南大），Java 生态成熟 |
| **OAuth 2.0 / OIDC** | 现代标准协议，OpenID Connect | 新系统推荐，微信/钉钉/飞书都是这个 |
| **自定义 SSO** | 各校自研的统一认证网关 | 如正方教务系统的统一身份认证 |

### 6.3 推荐方案：Spring Security OAuth2 Client

如果学校提供 **OAuth 2.0 / OIDC** 认证服务（大部分现代统一身份认证平台都支持），用 Spring Security 的 OAuth2 Client 集成是最简洁的方式。

#### 步骤 1：添加依赖

```xml
<!-- pom.xml 中添加 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-oauth2-client</artifactId>
</dependency>
```

#### 步骤 2：配置 OAuth2 客户端

```yaml
# application.yml
spring:
  security:
    oauth2:
      client:
        registration:
          campus-sso:
            client-id: ${CAMPUS_SSO_CLIENT_ID}        # 学校分配的 client_id
            client-secret: ${CAMPUS_SSO_CLIENT_SECRET} # 学校分配的密钥
            authorization-grant-type: authorization_code
            redirect-uri: "{baseUrl}/login/oauth2/code/campus-sso"
            scope: openid, profile                      # 请求的权限范围
        provider:
          campus-sso:
            issuer-uri: https://sso.school.edu.cn       # 学校 SSO 的 OIDC 发现地址
```

#### 步骤 3：Security 配置

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // OAuth2 登录
            .oauth2Login(oauth2 -> oauth2
                .loginPage("/login")
                .successHandler((request, response, authentication) -> {
                    // 获取校园认证返回的用户信息
                    OAuth2User oauthUser = (OAuth2User) authentication.getPrincipal();
                    String studentId = oauthUser.getAttribute("student_id"); // 或 "sub"
                    String studentName = oauthUser.getAttribute("name");
                    
                    // 写入 Session（与当前 AuthController 保持一致）
                    HttpSession session = request.getSession();
                    session.setAttribute("studentId", studentId);
                    session.setAttribute("studentName", studentName);
                    
                    // 如果学生不存在，自动创建
                    autoRegisterIfNeeded(studentId, studentName);
                    
                    response.sendRedirect("/");
                })
            )
            // 其他路径的权限控制
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/login", "/css/**", "/js/**").permitAll()
                .anyRequest().authenticated()
            );
        return http.build();
    }
    
    private void autoRegisterIfNeeded(String studentId, String studentName) {
        // 如果学生表中不存在，自动创建记录
        if (!studentRepository.existsById(studentId)) {
            StudentEntity student = new StudentEntity();
            student.setStudentId(studentId);
            student.setStudentName(studentName);
            student.setPassword(""); // OAuth 登录不需要密码
            studentRepository.save(student);
        }
    }
}
```

## 七、RAG 知识库与聊天记录运行说明

### RAG 知识库

**✅ 功能正常工作**，但有一个重要细节需要注意：

> ⚠️ **知识库索引存储在内存中（InMemoryEmbeddingStore），服务重启后会丢失！**

**每次服务重启后**，需要调用一次索引构建接口：
```
GET /ai/embedding-index
```

这会将 8 条寝室规则重新向量化并加载到内存。如果用户问规则相关问题（如"几点关门"）但没有执行索引构建，会返回「未找到相关规定」。

**如果需要持久化**（推荐），可以将 `InMemoryEmbeddingStore` 替换为：
- **Redis**（适合已有 Redis 的环境）
- **pgvector**（PostgreSQL 扩展，适合已有 PG 的环境）  
- **Chroma / Milvus**（专业向量数据库,更高级）

```java
// 替换方案示例：使用 Redis 存储向量
@Bean
public EmbeddingStore<TextSegment> embeddingStore() {
    return RedisEmbeddingStore.builder()
        .host("localhost")
        .port(6379)
        .dimension(1024)  // text-embedding-v3 是 1024 维
        .build();
}
```

### 聊天记录

**✅ 完全持久化**，存储在 MySQL 的 `chat_history` 表中。

- LLM 可通过 `ChatHistoryTools.getChatHistory()` 工具读取最近 20 条对话
- 前端通过 `/ai/chat-history?userId=xxx` 分页查询
- 用户可点击「清除对话」调用 `/ai/clear-chat-history/{userId}` 清空
- AOP 切面（`ChatFlowAop`）自动保存每次对话，无需手动调用
